# 16. 디버깅 완벽 가이드

> 오늘의 목표: 문제를 감으로 때리는 대신, 브레이크포인트와 호출 흐름을 이용해 정확하게 추적하는 습관을 만든다.

---

## 오늘 끝나면 되는 것

- Breakpoint, Step Over, Step Into, Step Out 차이를 설명할 수 있다.
- 변수 값과 Call Stack을 읽을 수 있다.
- 로그와 디버거를 언제 각각 쓰면 좋은지 구분할 수 있다.
- 400, 404, 500 문제를 추적하는 순서를 스스로 잡을 수 있다.

---

## 머릿속 그림

디버깅의 핵심은 이 질문입니다.

> "지금 어디까지 왔고, 여기서 값이 왜 이렇게 되었는가?"

그래서 디버깅은 단순 기능이 아니라,
흐름 추적 기술입니다.

---

## 가장 많이 쓰는 기능

### Breakpoint

코드를 해당 줄에서 멈추게 합니다.

### Step Over

현재 줄 실행 후 다음 줄로 이동합니다.

### Step Into

메서드 내부로 들어갑니다.

### Step Out

현재 메서드 실행을 마치고 호출한 곳으로 나옵니다.

---

## 어떤 상황에서 어디에 찍을까

### 400 Bad Request

- Controller 진입 전 막혔는가
- `@RequestBody` 역직렬화 실패인가
- Validation 실패인가

### 404 Not Found

- Handler Mapping이 안 되었는가
- 서비스에서 데이터가 없는가

### 500 Internal Server Error

- 서비스 로직 예외인가
- NPE인가
- 파일 경로/변환 문제인가

이렇게 에러 유형별로 디버깅 시작점이 달라집니다.

---

## Call Stack 읽는 습관

예외가 나면 메시지만 보지 말고 Call Stack을 보세요.

왜 중요하냐면:

- 어디서 호출됐는지
- 어떤 흐름을 타고 들어왔는지
- 프레임워크 내부에서 터진 건지, 내 코드에서 터진 건지

를 알 수 있기 때문입니다.

---

## 로그와 디버거는 경쟁 관계가 아니다

로그가 좋은 경우:

- 운영 환경
- 장시간 흐름 추적
- 비동기/원격 환경

디버거가 좋은 경우:

- 로컬 재현 가능
- 특정 요청 1건 깊게 파고들기
- 변수 상태 확인

둘을 같이 써야 가장 강합니다.

---

## 조건부 브레이크포인트

특정 조건에서만 멈추게 하면 반복 요청이나 루프 디버깅에 유용합니다.

예:

- `id == 10`
- `request.getTitle() == null`

이 기능 하나만 잘 써도 디버깅 효율이 꽤 올라갑니다.

---

## 자주 하는 실수

- 무작정 Step Into만 연타하는 것
- 현재 값보다 "예상값"만 떠올리는 것
- 로그와 예외 메시지를 안 읽고 바로 코드를 고치는 것
- 재현 조건을 정리하지 않고 디버깅하는 것
