## 04강. Hydration Deep Dive: 정적 HTML에 자바스크립트 생명력을 불어넣는 마법과 Mismatch 해결법

지난 시간 우리는 화면을 그리는 무거운 짐을 브라우저에서 서버로 옮겨오는 놀라운 과정, SSR(서버 사이드 렌더링)과 RSC(리액트 서버 컴포넌트)를 함께 살펴봤습니다. 서버에서 미리 완성된 화면(HTML)을 슝~ 하고 보내주니, 하얀 로딩 화면 없이 쾌적한 웹서핑이 가능해졌죠.

하지만 여기서 우리가 반드시 짚고 넘어가야 할 '불편한 진실'이 하나 있습니다. 서버에서 막 도착한 직후의 HTML은 사실 **'죽어있는 상태'** 와 다름없다는 것입니다. 화면에 버튼은 예쁘게 보이는데 아무리 눌러도 반응이 없고, 드롭다운 메뉴도 열리지 않습니다. 겉모습은 사람과 똑같지만 영혼이 없는 차가운 '밀랍 인형'인 셈이죠.

오늘 우리는 이 밀랍 인형에 영혼을 불어넣어 살아 움직이게 만드는 마법 같은 과정, **'하이드레이션(Hydration)'** 에 대해 깊이 파고들어 보겠습니다.

### 📖 개념 가이드 1: 도대체 '하이드레이션'이 뭔가요?

영어 단어가 낯설게 느껴질 수 있지만, 원리는 아주 간단합니다. Hydration의 사전적 의미는 **'수분 공급'** 입니다.

> 💡 **찰떡 비유: 컵라면 속 바짝 말린 건조 채소**
>
> 우리가 흔히 먹는 컵라면이나 인스턴트 국에 들어있는 건조 채소를 떠올려 보세요. 모양은 분명 파, 버섯, 고추인데 바짝 말라서(Dry) 씹어 먹을 수도 없고 맛도 나지 않습니다. 이것이 바로 서버에서 갓 도착한 '정적인 HTML'의 상태입니다.
>
> 이 딱딱한 채소를 먹을 수 있게 만들려면 무엇이 필요할까요? 바로 '뜨거운 물'입니다. 바짝 마른 정적인 HTML(건조 채소)에 자바스크립트(뜨거운 물)를 쫘악 부어주면, 채소가 촉촉해지며 생생하게 살아나듯 죽어있던 HTML이 클릭도 되고 입력도 되는 '살아있는 앱'으로 변신합니다.

이렇게 수분(자바스크립트)을 공급하여 생명력을 띠게 만드는 과정을 개발 용어로 **'하이드레이션'** 이라고 부릅니다.

### 📖 개념 가이드 2: 서버와 브라우저의 릴레이 협동 (포크와 나이프)

이 과정을 완벽하게 이해하기 위해, 지난 시간에 배웠던 렌더링 방식들과 연결해서 큰 그림을 그려봅시다.

- **CSR (이케아 가구):** 텅 빈 방에 덜렁 도착한 조립 설명서. 브라우저가 처음부터 끝까지 다 조립해야 해서 초기 로딩이 너무 느렸습니다.
- **SSR (고급 스테이크 하우스):** 주방장(서버)이 주방에서 스테이크를 완벽하게 구워서 접시에 담아 내어줍니다. 우리는 자리에 앉자마자 맛있는 음식을 눈으로 볼 수 있죠!

> 🚨 **그런데 문제가 생겼습니다.**
>
> 눈앞에 스테이크(완성된 HTML)는 있는데, 썰어 먹을 포크와 나이프(자바스크립트/이벤트 리스너)가 도착하지 않은 겁니다! 상호작용을 할 수 없는 답답한 찰나의 순간이 발생합니다.
>
> 이때 웨이터(브라우저)가 달려와 여러분의 손에 포크와 나이프를 착착 쥐여주는 과정, 뼈대에 알맞게 이벤트 리스너를 끼워 맞춰 생명력을 불어넣는 과정! 이것이 바로 **하이드레이션**입니다.

### 📖 개념 가이드 3: Next.js의 필살기, '부분 하이드레이션'

모든 프레임워크가 하이드레이션을 똑같이 하는 것은 아닙니다. 과거에는 아주 작은 '좋아요' 버튼 하나를 살리기 위해, 페이지 전체에 거대한 양의 물(자바스크립트)을 들이부어야 했습니다. 비효율적이죠.

하지만 Next.js는 **'부분 하이드레이션(Partial Hydration)'** 이라는 놀라운 기술을 씁니다.

> 💡 **찰떡 비유: 똑똑한 스마트 스프링클러**
>
> 마당에 물을 줄 때, 생명이 없는 돌멩이(정적인 텍스트)에는 물을 주지 않고, 오직 생명이 피어날 꽃(버튼, 입력창 등)에만 정확하게 물을 쏘아주는 스마트 스프링클러와 같습니다.

지난 시간 배운 RSC(서버 컴포넌트)는 영원히 변하지 않는 돌멩이이므로 아예 하이드레이션 대상에서 제외됩니다. 오직 `'use client'` 출입증이 붙은 클라이언트 컴포넌트(꽃)만 콕 집어서 물을 줍니다. 덕분에 Next.js 앱은 획기적으로 가볍고 빠릅니다!

### 🚨 개념 가이드 4: 전 세계 개발자의 악몽, Hydration Mismatch

이 정교한 협동 작업에는 아주 까다롭고 무서운 규칙이 하나 숨어있습니다.

> **"서버(주방장)가 내어준 요리의 모양과, 브라우저(웨이터)가 예상한 요리의 모양이 단 1밀리미터, 1픽셀도 틀리지 않고 완벽하게 일치해야 한다!"**

만약 주방장은 '스테이크'를 보냈는데, 웨이터가 '수프'인 줄 알고 숟가락을 가져다준다면 식당은 난장판이 되겠죠? 리액트도 마찬가지입니다. 서버에서 그린 그림과 브라우저가 그리려는 그림이 다르면, 리액트는 혼란에 빠져 기존 HTML을 다 부수고 처음부터 다시 그리는 비효율적인 짓을 하며 빨간 에러를 토해냅니다.

이것이 바로 **Hydration Mismatch(불일치)** 현상입니다.

> ✨ **Next.js 15의 기쁜 소식!**
>
> 과거에는 이 에러 메시지가 정말 불친절해서 '악몽' 같았는데, 버전 15부터는 서버와 브라우저의 차이를 마치 '틀린 그림 찾기' 게임처럼 아주 명확하게 비교해서 보여줍니다. 덕분에 원인을 찾기가 훨씬 쉬워졌죠. 하지만 여전히 성능에는 치명적이니 꼭 해결해야겠죠?

우리가 무의식적으로 어떤 실수를 하는지 코드로 볼까요?

```tsx
/* ❌ [안티 패턴] Mismatch를 유발하는 전형적인 초보자의 실수 */
export default function TimeDisplay() {

  // 🚨 실수 1: 시간 (Time)
  // 서버가 HTML을 만든 시간(10:00:00)과,
  // 인터넷 선을 타고 브라우저에 도착해 JS가 실행된 시간(10:00:01)은 영원히 다릅니다!
  const time = new Date().toLocaleTimeString();

  // 🚨 실수 2: 랜덤 값 (Random)
  // 서버가 굴린 주사위 값(0.12)과 브라우저가 굴린 주사위 값(0.99)은 일치할 수 없습니다!
  const randomId = Math.random();

  return (
    <div id={randomId.toString()}>
      <h1>현재 시각</h1>
      <p>{time}</p>

      {/* 🚨 실수 3: 잘못된 HTML 구조 (Invalid Nesting) */}
      <p>
        {/* p태그(문단) 안에는 div태그(구역)가 들어갈 수 없습니다. (HTML 문법 위반) */}
        {/* 브라우저가 이걸 몰래 고치려고 시도하다가 리액트에게 딱 걸려서 에러가 납니다. */}
        <div>Description</div>
      </p>
    </div>
  );
}
```

### 🛠️ 개념 가이드 5: Mismatch를 우아하게 해결하는 두 가지 마법

이 무서운 에러를 해결하려면 **"완벽하게 동기화"** 하거나, **"알고도 눈감아주기"** 스킬을 써야 합니다.

```tsx
/* ✅ [해결책] Mismatch를 해결하는 올바른 패턴의 정석 */
'use client';
import { useState, useEffect } from "react";

export default function SafeTimeDisplay() {
  // ✨ 패턴 1: Two-Pass Rendering (강제 일치시키기)
  // 초기값을 아예 비워둡니다(null). 서버도 "없음!", 브라우저의 첫 모습도 "없음!"으로 일치시킵니다.
  const [time, setTime] = useState<string | null>(null);

  useEffect(() => {
    // useEffect는 '하이드레이션(수분 공급)이 완전히 끝난 후'에만 실행됩니다.
    // 즉, "포크와 나이프가 다 지급된 후(오직 브라우저에서만!)" 시간을 세팅합니다.
    setTime(new Date().toLocaleTimeString());
  }, []);

  return (
    <div>
      <h1>현재 시각</h1>

      {/* 해결 방법 1: 로딩 중 처리 (안전함의 정석) */}
      {time ? <p>{time}</p> : <p>Loading...</p>}

      {/* ✨ 패턴 2: suppressHydrationWarning (치트키: 알고도 눈감아주기) */}
      {/* "리액트야, 서버랑 브라우저 시간 다른 거 나도 알아. 에러 띄우지 말고 그냥 브라우저 값으로 덮어씌워!" */}
      {/* ⚠️ 주의: 텍스트가 다를 때만 써야지, 태그 구조 자체가 다를 때 쓰면 앱이 망가집니다! */}
      <p suppressHydrationWarning>
        {new Date().toLocaleTimeString()}
      </p>
    </div>
  );
}
```

| 패턴 | 설명 |
|---|---|
| **패턴 1 (Two-Pass Rendering)** | 서버와 브라우저의 첫인상을 강제로 일치시키고, 브라우저가 준비된 이후(`useEffect`)에 데이터를 넣는 가장 안전한 정석입니다. |
| **패턴 2 (suppressHydrationWarning)** | 어쩔 수 없이 텍스트 차이가 날 때 사용하는 리액트 전용 치트키입니다. |

### 🚀 마무리하며

오늘 우리는 차갑고 정적인 HTML이 자바스크립트라는 물(수분)을 머금고 살아 움직이는 생명체로 변신하는 하이드레이션(Hydration)의 신비를 풀어냈습니다. 서버와 클라이언트의 결과물을 동기화하는 것이 얼마나 중요하고 예민한 작업인지도 깨달으셨을 겁니다.

이제 우리 앱은 서버에서 눈 깜짝할 새 날아와서, 브라우저에서 부드럽게 살아 움직입니다! 하지만 아직 한 가지 거대한 퍼즐이 남았습니다. 우리가 일반적인 웹사이트에서 다른 페이지로 이동(링크 클릭)하면 화면이 하얗게 껌뻑! 하고 깜빡입니다. 그런데 Next.js로 만든 사이트는 페이지를 이동해도 앱처럼 부드럽게 넘어가며 화면이 전혀 깜빡이지 않습니다.

도대체 Next.js의 `<Link>` 컴포넌트는 어떤 마술을 부리는 걸까요?

## 05강. Navigation Experience: 새로고침(Hard)의 한계와 완제품을 렌더링하는 Soft Navigation

우리는 지난 4개의 강의를 통해 Next.js의 탄생 배경부터 렌더링, 그리고 하이드레이션까지 숨 가쁘게 달려왔습니다. 드디어 오늘, 그 모든 이론이 집대성되어 사용자에게 직접적으로 느껴지는 경험, 바로 **'네비게이션(Navigation)'** 을 다룰 차례입니다.

유튜브나 넷플릭스를 볼 때 페이지를 넘겨도 영상이 끊기지 않고 부드럽게 이어지는 것을 보셨나요? 마치 웹사이트가 아니라 스마트폰 앱을 쓰는 것 같은 이 매끄러운 경험. 이것이 바로 오늘 우리가 파헤칠 **Soft Navigation**의 정체입니다.

### 1. Hard Navigation의 한계: '철거 후 재건축'의 비효율

Next.js 도입 전, 우리가 전통적인 `<a>` 태그를 사용했던 시절은 Hard Navigation의 시대였습니다. 이것은 웹의 탄생과 함께한 가장 원시적인 방식입니다.

> 💡 **비유: 이사 (포크레인 철거)**
>
> 기존 방식의 페이지 이동은 옆집으로 이사를 가기 위해 살던 집을 포크레인으로 완전히 부수고, 새 땅에 기초 공사부터 다시 시작하는 것과 같습니다. 브라우저는 현재 페이지를 파괴(Unmount)하고, 서버로부터 새로운 HTML을 받아 처음부터 다시 그립니다.

> 🔍 **하드 네비게이션의 3대 페인 포인트 (Pain Points)**
>
> 1. **화면 깜빡임 (White Flash):** 집이 부서지고 새 집이 지어지는 찰나의 순간, 사용자는 하얀 공백 화면을 마주해야 합니다.
> 2. **상태 소멸 (State Loss):** 열심히 작성하던 댓글, 스크롤 위치, 열어둔 사이드바 등 메모리에 있던 모든 정보가 '새로고침'과 함께 증발합니다.
> 3. **비효율:** 공통으로 쓰는 헤더나 푸터(Header/Footer)까지 모두 부수고 다시 그립니다.

```tsx
/* [Pain Point]: Hard Navigation (HTML 기본 태그) - 파괴적인 이동 */

export default function NavBar() {
  return (
    <nav>
      {/* 클릭하는 순간 브라우저가 '새로고침' 되며 모든 상태가 초기화됩니다. */}
      {/* 마치 집을 부수고 다시 짓는 것과 같습니다. */}
      <a href="/dashboard">Dashboard (Hard)</a>
      <a href="/settings">Settings (Hard)</a>
    </nav>
  );
}
```

### 2. Solution: Soft Navigation과 '완제품 가구' 배송

Next.js는 이 문제를 `<Link>` 컴포넌트를 통한 **Soft Navigation**으로 해결했습니다.

> 🔑 **핵심 개념: RSC Payload (부분 리모델링)**
>
> 일반적인 리액트(SPA)가 '목재와 나사(JSON)'만 보내줘서 브라우저가 직접 조립해야 했다면, Next.js는 공장에서 이미 조립된 **'완제품 가구(RSC Payload)'** 를 보내줍니다. 브라우저는 "거실 벽지는 그대로 두고, 소파만 이걸로 바꿔!"라는 지시를 받고, 필요한 부분만 '톡' 하고 갈아 끼웁니다.

| 방식 | 비유 | 동작 |
|---|---|---|
| **React Router** | 텅 빈 방에 목재(JSON)와 설명서(JS)가 도착 | 브라우저가 땀 흘려 가구를 조립해야 화면이 뜹니다. (클라이언트 연산 부담 ⬆️) |
| **Next.js Router** | 완성된 가구(RSC Payload)가 도착 | 브라우저는 배달 온 가구를 제자리에 놓기만 하면 됩니다. (브라우저 부담 ⬇️, 속도 ⬆️) |

```tsx
/* [Solution]: Next.js <Link> (Soft Navigation) - 부드러운 전환 */
import Link from "next/link";

export default function NavBar() {
  return (
    <nav>
      {/* 클릭 시 서버에서 '완제품(RSC Payload)'만 받아와 즉시 교체합니다. */}
      {/* 새로고침이 발생하지 않아 스크롤, 영상 재생 상태가 유지됩니다. */}
      <Link href="/dashboard">Dashboard (Soft)</Link>
      <Link href="/settings">Settings (Soft)</Link>
    </nav>
  );
}
```

### 3. Deep Dive: '프리패칭(Prefetching)'이라는 마법

Next.js의 `<Link>`에는 또 하나의 숨겨진 무기가 있습니다. 바로 **미리 가져오기(Prefetching)** 입니다.

> ⚡ **동작 원리: 셰프의 예측 요리**
>
> 여러분이 식당에서 메뉴판의 '파스타'를 뚫어져라 쳐다보고 있다고 상상해 보세요. 눈치 빠른 셰프(Next.js)는 여러분이 주문하기도 전에 주방에서 이미 면을 삶기 시작합니다.

1. **감지 (Viewport):** 링크가 사용자의 화면(눈)에 들어오는 순간, Next.js가 감지합니다.
2. **다운로드 (Background):** 사용자가 클릭도 하기 전에, 백그라운드에서 다음 페이지의 데이터를 몰래 받아옵니다.
3. **즉시 전환 (Instant Transition):** 사용자가 실제로 클릭했을 때, 이미 요리는 끝났습니다. 네트워크 대기 시간 없이 0초 만에 화면이 전환됩니다.

## 면접 대비 핵심 문제 (Part 2)

### Q1. 하이드레이션(Hydration)이란 무엇이며, 왜 필요한지 설명해주세요.

<details>
<summary>모범 답안</summary>

**하이드레이션(Hydration)** 이란 서버에서 렌더링되어 전달된 정적 HTML에 자바스크립트 이벤트 리스너를 연결하여, 상호작용이 가능한 동적인 앱으로 만드는 과정입니다.

**왜 필요한가:**
- SSR/RSC로 서버에서 전송된 HTML은 눈에 보이지만 클릭, 입력 등의 상호작용이 불가능한 "죽어있는 상태"입니다.
- 하이드레이션을 통해 React가 기존 DOM 구조를 재사용하면서 이벤트 핸들러를 부착합니다.
- 이 과정이 완료되어야 비로소 사용자가 버튼 클릭, 폼 입력 등을 할 수 있습니다.

**핵심 포인트:** 하이드레이션은 처음부터 DOM을 새로 생성하는 것이 아니라, 서버가 만들어준 기존 DOM을 "입양(adopt)"하고 그 위에 이벤트를 입히는 과정입니다.

</details>

### Q2. Next.js의 부분 하이드레이션(Partial Hydration)이 기존 방식보다 효율적인 이유를 설명해주세요.

<details>
<summary>모범 답안</summary>

**기존 방식 (Full Hydration):**
- 페이지 전체에 자바스크립트를 적용하여 하이드레이션을 수행합니다.
- 정적 텍스트처럼 상호작용이 불필요한 부분까지 모두 하이드레이션 대상이 됩니다.
- 불필요한 JS 전송량 증가 및 하이드레이션 시간 지연이 발생합니다.

**Next.js의 부분 하이드레이션 (Partial Hydration):**
- **서버 컴포넌트(RSC)는 하이드레이션 대상에서 완전히 제외**됩니다.
- 오직 `'use client'` 지시어가 선언된 클라이언트 컴포넌트만 선택적으로 하이드레이션합니다.
- 결과적으로 브라우저로 전송되는 JS 번들이 크게 줄어들고, 하이드레이션 소요 시간도 단축됩니다.

이는 "상호작용이 필요한 꽃에만 물을 주고, 돌멩이에는 물을 주지 않는 스마트 스프링클러"와 같은 원리입니다.

</details>

### Q3. Hydration Mismatch가 발생하는 원인 3가지와 해결 방법을 설명해주세요.

<details>
<summary>모범 답안</summary>

**Hydration Mismatch란:** 서버에서 렌더링한 HTML과 클라이언트에서 렌더링하려는 결과물이 불일치할 때 발생하는 에러입니다. React는 혼란에 빠져 기존 HTML을 폐기하고 처음부터 다시 렌더링하므로 성능이 크게 저하됩니다.

**발생 원인 3가지:**

1. **시간 의존 값:** `new Date()`는 서버 실행 시점과 클라이언트 실행 시점이 다르므로 결과가 불일치합니다.
2. **랜덤 값:** `Math.random()`은 서버와 클라이언트에서 서로 다른 값을 생성합니다.
3. **잘못된 HTML 구조:** `<p>` 안에 `<div>`를 넣는 등 HTML 명세 위반 시, 브라우저가 자동으로 DOM을 보정하면서 서버 결과와 차이가 발생합니다.

**해결 방법:**

| 패턴 | 방법 | 적용 상황 |
|---|---|---|
| **Two-Pass Rendering** | 초기값을 `null`로 설정 후, `useEffect`에서 클라이언트 전용 값을 세팅 | 시간, 랜덤 값 등 서버/클라이언트 결과가 다를 수밖에 없는 경우 (가장 안전한 정석) |
| **suppressHydrationWarning** | JSX에 `suppressHydrationWarning` 속성 추가 | 텍스트 내용만 미세하게 다른 경우에 한정 (태그 구조가 다르면 사용 불가) |

</details>

### Q4. Hard Navigation과 Soft Navigation의 차이를 설명하고, Next.js에서 Soft Navigation이 동작하는 원리를 말해주세요.

<details>
<summary>모범 답안</summary>

**Hard Navigation (`<a>` 태그):**
- 브라우저가 현재 페이지를 완전히 파괴(Unmount)하고, 서버에 새로운 HTML 문서를 요청합니다.
- 전체 페이지가 새로고침되므로 화면 깜빡임(White Flash)이 발생합니다.
- 클라이언트 상태(스크롤 위치, 입력값, 컴포넌트 상태 등)가 모두 소멸합니다.

**Soft Navigation (`<Link>` 컴포넌트):**
- 브라우저의 새로고침 없이, 변경이 필요한 영역의 RSC Payload만 서버에서 받아와 교체합니다.
- 공통 레이아웃(Header, Footer 등)은 유지되고 변경된 부분만 갱신됩니다.
- 클라이언트 상태가 보존되어 앱과 같은 부드러운 사용자 경험을 제공합니다.

**동작 원리:**
1. `<Link>`를 클릭하면 Next.js 클라이언트 라우터가 이벤트를 가로챕니다 (`e.preventDefault()`).
2. History API를 사용하여 URL을 변경합니다.
3. 서버에 해당 경로의 RSC Payload를 요청합니다.
4. 받아온 Payload로 변경된 영역만 DOM을 업데이트합니다.

</details>

### Q5. Next.js `<Link>` 컴포넌트의 Prefetching은 어떻게 동작하며, 성능에 어떤 영향을 주나요?

<details>
<summary>모범 답안</summary>

**Prefetching 동작 방식:**

1. **Viewport 감지:** `<Link>` 컴포넌트가 사용자의 뷰포트(화면)에 들어오면, Intersection Observer를 통해 감지합니다.
2. **백그라운드 다운로드:** 사용자가 클릭하기 전에, 해당 링크가 가리키는 페이지의 RSC Payload와 필요한 JS 번들을 백그라운드에서 미리 다운로드합니다.
3. **즉시 전환:** 실제 클릭 시 이미 데이터가 준비되어 있으므로 네트워크 대기 시간 없이 즉시 페이지가 전환됩니다.

**성능 영향:**
- **체감 로딩 시간이 0에 가까워집니다** (네트워크 지연이 사라짐).
- 사용자가 실제로 방문하지 않을 페이지도 미리 받을 수 있어, 대역폭 사용량이 증가할 수 있습니다.
- `prefetch={false}` 옵션으로 비활성화하거나, 프로덕션 환경에서만 동작하는 등 최적화가 적용되어 있습니다.

</details>

### Q6. CSR → SSR → RSC로의 렌더링 패러다임 변화를 "브라우저 부담"의 관점에서 정리해주세요.

<details>
<summary>모범 답안</summary>

프론트엔드 렌더링의 역사는 **"화면을 그리는 무거운 짐을 누가 질 것인가"** 에 대한 줄다리기입니다.

| 세대 | 방식 | 브라우저 부담 | 설명 |
|---|---|---|---|
| 1세대 | **CSR** | 매우 높음 | 빈 HTML + 거대한 JS 번들을 받아 브라우저가 모든 렌더링과 데이터 요청을 처리합니다. |
| 2세대 | **SSR** | 중간 | 서버가 HTML을 미리 완성해서 보내지만, 상호작용을 위한 거대한 JS 번들은 여전히 브라우저로 전송됩니다. |
| 3세대 | **RSC** | 최소 | 서버 로직의 JS는 아예 브라우저로 전송되지 않습니다. `'use client'` 컴포넌트의 최소한의 JS만 전달됩니다. |

**핵심 흐름:**
- CSR: 모든 짐을 브라우저가 짊어짐 (번들 크기 최대)
- SSR: 초기 렌더링은 서버가 대신하지만, 하이드레이션을 위한 JS는 여전히 전송
- RSC: 서버 코드의 JS 전송량이 0 (Zero Bundle Size), 부분 하이드레이션으로 최소한의 JS만 전송

이 진화를 통해 **초기 로딩 속도, SEO, 사용자 경험** 모두가 개선되었습니다.

</details>
