## 01강. Library vs Framework: 제어의 역전(IoC)과 Next.js의 탄생 배경

리액트(React)라는 바다를 건너온 여러분, 환영합니다. 이제 우리는 그 위에 견고한 웹 서비스라는 성을 쌓아 올릴 Next.js라는 거대한 대륙에 도착했습니다.

본격적인 문법 학습에 앞서, 우리가 왜 리액트만으로는 충분하지 않았는지, 그리고 Next.js가 우리에게 쥐여주는 가장 강력한 무기가 무엇인지 그 근본적인 아키텍처와 철학을 파헤쳐 봅니다.

### 1. Library의 한계: '빈 땅'의 건축가와 설정(Configuration) 지옥

과거 순수 리액트(Library)만으로 프로젝트를 구축하던 시절, 개발자는 마치 '빈 땅을 산 건축가'와 같았습니다.

> 💡 **비유: 셀프 건축 시스템**
> 리액트는 화면을 그리는 'UI'라는 벽돌만 제공할 뿐입니다. 건물을 짓기 위해서는 기둥(라우터), 시멘트(상태 관리), 전기 배선(번들러) 등 수십 가지 도구를 개발자가 직접 선택하고 조립해야 했습니다.

이러한 무한한 자유는 곧 '설정(Configuration)의 늪'이라는 피로감으로 다가왔습니다. 페이지 이동 하나를 위해 우리가 감내해야 했던 React Router의 설정 코드를 살펴봅시다.

```tsx
/* [Pain Point]: React(Library) - 개발자가 모든 것을 수동으로 지시하는 방식 */
import { BrowserRouter, Routes, Route, Link } from "react-router-dom";

function App() {
  return (
    // 1. 라우터 껍데기를 씌우고
    <BrowserRouter>
      <Routes>
        {/* 2. 어떤 URL에 어떤 화면을 보여줄지 개발자가 일일이 '매핑'합니다. */}
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
      </Routes>
    </BrowserRouter>
  );
}
```

> 🔍 **라이브러리 방식의 3대 페인 포인트 (Pain Points)**
>
> 1. **높은 피로도:** 필요한 외부 도구를 일일이 설치하고, 그 도구의 규칙을 새로 학습해야 합니다.
> 2. **유지보수의 어려움:** 페이지가 하나 늘어날 때마다 중앙 설정 파일(Entry Point)을 열어 라우팅 코드를 수정해야 하는 막노동이 발생합니다.
> 3. **주도권의 피곤함:** 코드를 '호출'하고 '제어'하는 모든 책임과 주도권이 개발자인 '나'에게 있습니다.

### 2. Framework의 등장: '제어의 역전(IoC)'이라는 패러다임 시프트

이제 Next.js의 세계로 넘어오면 상황은 완전히 뒤바뀝니다. 프레임워크는 빈 땅이 아니라 골조와 배관이 모두 갖춰진 '풀옵션 모델하우스'입니다.

이 거대한 변화를 관통하는 단 하나의 핵심 컴퓨터 공학 원리가 바로 **제어의 역전(Inversion of Control, IoC)** 입니다.

> 🔑 **핵심 개념: 제어의 역전 (IoC) 이란?**
> 쉽게 말해 '주도권이 넘어갔다'는 뜻입니다. 과거에는 내 코드가 중심에 있고 내가 도구들을 호출했다면, 이제는 프레임워크(Next.js)가 뼈대를 잡고 있고 내 코드는 그저 부품처럼 끼워져 '호출 당하는' 구조가 된 것입니다.

> 💡 **비유: Next.js 호텔 시스템**
> Next.js에서 `app` 폴더는 호텔의 '로비'이고, 그 안의 폴더들은 '방'입니다. 우리가 만들 `page.tsx`는 그 방의 메인 공간이죠. "101호 문을 열면 침대를 보여줘"라고 지시서를 쓸 필요 없이, 101호 문을 여는 순간 프레임워크가 알아서 방을 보여줍니다.

```tsx
/* [Solution]: Next.js(Framework) - 설정 대신 '관습'을 따르는 마법 */

// 1. app/page.tsx (메인 화면)
export default function HomePage() {
  return <h1>🏨 호텔 로비 (/)</h1>;
}

// 2. app/about/page.tsx (About 화면)
export default function AboutPage() {
  // 라우터 설정 코드 없이, 폴더와 파일 생성만으로 '/about' 주소가 연결됩니다.
  return <h1>ℹ️ 101호 객실 (/about)</h1>;
}
```

**제어의 역전이 가져온 결과:** 개발자는 `export default`를 통해 완성된 요리(컴포넌트)를 주방장(Next.js)에게 건네주기만 하면 됩니다. 렌더링을 언제, 어떻게 할 것인지는 전적으로 프레임워크가 제어합니다.

### 3. 결론: 설정(Configuration)보다 관습(Convention)

Next.js를 배운다는 것은 단순한 문법 공부가 아닙니다. "설정보다 관습을 따른다"는 프레임워크의 철학에 동의하는 과정입니다.

| 구분 | 특징 |
|---|---|
| **Library (React)** | 내가 도구를 조종합니다. (자유도 ⬆️, 설정 피로도 ⬆️) |
| **Framework (Next.js)** | 프레임워크가 나를 조종합니다. 나는 규칙만 지킵니다. (생산성 ⬆️, 비즈니스 로직 집중 ⬆️) |

프레임워크가 닦아놓은 고속도로의 규칙만 잘 지켜주면, 내부 엔진이 복잡한 배선과 라우팅 최적화를 완벽하게 처리해 줍니다. 특히, 우리가 앞으로 다룰 이 `app` 폴더 기반의 아키텍처는 Next.js 15의 가장 강력하고 완벽한 최신 표준인 **'App Router'** 방식입니다.

이제 우리는 "어떻게 연결할까?"라는 낡은 인프라 고민에서 완벽히 해방되어, "어떤 화면을 보여줄까?"라는 서비스의 본질에만 100% 집중할 수 있게 되었습니다.

## 02강. Routing Evolution: 수동 지도의 늪에서 벗어나는 '파일 시스템 라우팅'의 진화

"라우팅"이라는 말이 어렵게 들리시나요? 사실 아주 간단합니다. 여러분이 주소창에 `www.naver.com/news`라고 쳤을 때, "뉴스 화면을 보여줘!"라고 연결해 주는 교통정리 과정이 바로 라우팅입니다.

이 교통정리 방식이 과거(React Router)와 현재(Next.js) 어떻게 달라졌는지, 왜 개발자들이 Next.js에 열광하는지 아주 쉽게 풀어드리겠습니다.

### 📖 개념 가이드 1: 수동 지도의 늪 (도입 전)

> *"모든 길을 수첩에 적어야 했던 시절"*

Next.js를 쓰기 전에는, 페이지 하나를 만들 때마다 개발자가 직접 '지도'를 그려야 했습니다.

```tsx
/* [과거의 고통]: 하나하나 직접 연결해야 함 */

// 1. 페이지를 파일에서 가져옵니다. (import)
import Home from './Home';
import About from './About';
import Mypage from './Mypage';

// 2. 주소(URL)와 페이지(Component)를 짝지어줍니다.
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/about" element={<About />} />
  <Route path="/mypage" element={<Mypage />} />
</Routes>
```

> 🔍 **무엇이 문제였나요?**
>
> - **깜빡하면 에러:** 페이지 파일을 아무리 잘 만들어도, 저 `Routes` 목록에 등록하는 걸 깜빡하면 화면이 안 뜹니다. (마치 건물을 지어놓고 지도에 표시를 안 해서 아무도 못 찾아가는 꼴이죠.)
> - **오타의 지옥:** `/about`을 `/abput`이라고 실수로 적으면? 사용자는 영문도 모른 채 "페이지를 찾을 수 없습니다(404)"라는 에러만 보게 됩니다.
> - **관리의 어려움:** 페이지가 100개가 되면, 저 목록도 100줄이 넘어가서 알아보기가 힘들어집니다.

### 📖 개념 가이드 2: 파일 시스템 라우팅 (도입 후)

> *"건물을 지으면 주소가 자동으로 생기는 마법"*

Next.js는 이 모든 귀찮은 과정을 없앴습니다. **"폴더의 위치가 곧 주소다"** 라는 규칙 하나만 기억하면 됩니다.

```
/* [현재의 혁명]: Next.js 폴더 구조 */

app/
├── page.tsx               👉 인터넷 주소창: "내사이트.com/"
├── about/
│   └── page.tsx           👉 인터넷 주소창: "내사이트.com/about"
├── login/
│   └── page.tsx           👉 인터넷 주소창: "내사이트.com/login"
```

> 🔍 **핵심 포인트**
>
> - **설정 파일 삭제:** "이 주소로 오면 저 페이지 보여줘"라고 적던 수첩이 사라졌습니다.
> - **직관적:** 폴더 구조만 쓱 훑어봐도 "아, 이 사이트는 소개(about)랑 로그인(login) 페이지가 있구나" 하고 바로 알 수 있습니다.
> - **자동화:** 폴더를 만들고 `page.tsx` 파일을 넣는 순간, 전 세계 어디서든 접근 가능한 주소가 생성됩니다.

### 📖 개념 가이드 3: 만능 열쇠 `[id]` (동적 라우팅)

> *"상품이 1억 개라도 페이지는 하나면 됩니다"*

쇼핑몰을 만든다고 상상해 보세요. 상품이 1억 개입니다. 그럼 폴더를 1억 개 만들어야 할까요? `app/product/1`, `app/product/2`, `app/product/3` ...? 절대 아닙니다.

이때 사용하는 것이 바로 **대괄호 `[]`** 입니다.

```
app/
└── products/
    └── [id]/      👈 "여기는 어떤 단어든 들어올 수 있어!" (만능 폴더)
        └── page.tsx
```

> 🔍 **시뮬레이션: 사용자가 접속할 때 일어나는 일**
>
> 1. 사용자가 주소창에 `내사이트.com/products/galaxy`라고 입력합니다.
> 2. Next.js가 `products` 폴더로 갑니다.
> 3. 그 안에 `galaxy`라는 폴더는 없지만, `[id]`라는 만능 폴더가 있네요!
> 4. Next.js가 말합니다. "자, `[id]` 폴더야. 사용자가 방금 `galaxy`라고 입력했어. 이 단어를 `params`라는 봉투에 담아줄 테니 화면에 띄워!"
> 5. 페이지는 봉투를 열어보고 "아, 갤럭시 상품을 보여달라는구나!" 하고 알아챕니다.

### 💡 비유: 호텔 프런트 vs 아파트 호수

이 차이를 가장 쉽게 이해하는 비유입니다.

| 방식 | 비유 | 설명 |
|---|---|---|
| **과거 (수동 라우팅)** | 호텔 프런트 | 손님(사용자)이 오면 프런트 직원이 장부를 뒤적이며 안내해야 합니다. 실수로 장부에 방을 안 적어두면, 손님은 방을 못 찾습니다. |
| **현재 (파일 시스템 라우팅)** | 아파트 호수 | 별도 안내가 필요 없습니다. '101동 202호'라는 주소(폴더 경로)만 있으면 배달원(브라우저)이 정확히 찾아갑니다. |
| **동적 라우팅 `[id]`** | 빈 명찰 | "방문객"이라고 적힌 빈 명찰입니다. 철수가 오면 '철수'가 되고, 영희가 오면 '영희'가 되는 만능 명찰입니다. |

### 📝 마무리 (Wrap-up)

오늘 배운 내용을 한 줄로 요약하면 이렇습니다.

> **"Next.js에서는 폴더를 만드는 것이 곧 길을 닦는 것이다."**

이제 더 이상 복잡한 설정 파일과 씨름하지 마세요. 여러분은 그저 어떤 페이지(폴더)를 만들지만 고민하면 됩니다. 연결은 Next.js가 알아서 해주니까요.

## 03강. Rendering Mechanics: CSR의 폭포수를 넘어서는 RSC(서버 컴포넌트)의 혁명

프론트엔드 개발의 역사는 한마디로 **"이 화면을 그리는 무거운 짐을 서버가 질 것인가, 아니면 브라우저가 질 것인가?"** 에 대한 치열한 줄다리기였습니다.

### 📖 개념 가이드 1: 1세대 패러다임, CSR (Client Side Rendering)

가장 먼저 우리가 순수 리액트(React)만으로 개발하던 도입 전의 CSR 시절을 떠올려 봅시다. 여기서 'Client(클라이언트)'는 바로 여러분이 접속하는 크롬, 사파리 같은 **'브라우저'** 를 의미합니다. 즉, 브라우저가 화면을 그리는 모든 짐을 짊어졌던 시대입니다.

> 💡 **찰떡 비유: 조립되지 않은 이케아 가구 박스**
>
> 사용자가 사이트에 접속하면, 서버(가구 공장)는 완성된 가구가 아니라 텅 빈 상자(빈 HTML 파일)와 엄청나게 두꺼운 조립 설명서(거대한 자바스크립트 파일)만 덩그러니 던져줍니다.
>
> 브라우저(고객)는 그때부터 땀을 뻘뻘 흘리며 두꺼운 설명서를 다운로드해서 다 읽어봅니다. 다 읽고 났더니 "어? 나사(데이터)가 없네?" 하고 다시 나사를 찾기 위해 서버에 API 요청을 보냅니다. 나사가 도착하고 나서야 비로소 가구를 하나하나 조립(화면 렌더링)하기 시작합니다.

> 🚨 **CSR의 치명적인 고통: 폭포수(Waterfall) 현상**
>
> 이 과정에서 사용자는 빈 상자만 쳐다보며, 하얀 화면과 뱅글뱅글 도는 로딩 스피너를 하염없이 바라봐야 했습니다. 구글 검색 봇(SEO) 역시 빈 페이지만 보고 "이 사이트는 내용이 없네" 하고 떠나버리기 일쑤였죠.

```tsx
/* [Pain Point]: 초보자들이 가장 많이 겪는 CSR의 폭포수 지연 늪 */
import { useState, useEffect } from 'react';

export default function ProductPage() {
  // 1. 초기 상태: 처음엔 데이터가 없으니 빈 그릇(null)을 만듭니다. 화면엔 아무것도 없습니다.
  const [data, setData] = useState(null);
  const [isLoading, setIsLoading] = useState(true);

  // 2. 폭포수 지연 시작!
  // 화면이 텅 빈 채로 한 번 켜진 직후(useEffect)에야 서버로 데이터를 가지러 떠납니다.
  useEffect(() => {
    fetch('/api/products')
      .then((res) => res.json())
      .then((data) => {
        setData(data);
        setIsLoading(false); // 이 긴 통신이 끝나야 비로소 로딩 글자가 사라집니다.
      });
  }, []);

  // 3. 데이터가 오기 전까지 사용자가 가장 먼저 마주하는 불친절한 화면
  if (isLoading) return <div>데이터를 가져오는 중입니다... 빙글빙글🌀</div>;

  // 4. 아주 긴 기다림 끝에 드디어 상품 이름이 화면에 그려집니다.
  return <div>{data.name}</div>;
}
```

### 📖 개념 가이드 2: 2세대 과도기, SSR (Server Side Rendering)

"브라우저가 너무 고생하니까, 서버에서 화면을 미리 좀 그려서 주자!" 이러한 문제를 해결하기 위해 등장한 것이 바로 SSR입니다.

> 💡 **찰떡 비유: 랩으로 꽁꽁 싸인 완성형 가구**
>
> 이번엔 서버(가구 공장)의 전문가가 가구를 뼈대까지 다 조립해서 배달해 줍니다. 덕분에 브라우저(고객)는 상자를 열자마자 완성된 가구(미리 그려진 HTML 화면)를 즉시 눈으로 볼 수 있습니다. 하얀 로딩 화면의 고통이 사라진 것이죠!
>
> 하지만 맹점이 있었습니다. 가구에 흠집이 나지 않게 랩으로 꽁꽁 싸여 있어서, 눈으로 볼 수는 있지만 당장 서랍을 열거나 스위치를 누를 수(상호작용)는 없었습니다. 결국 기능을 활성화하려면, 랩을 벗기는 방법이 적힌 여전히 거대한 자바스크립트 설명서를 브라우저로 또 다운로드해야 한다는 근본적인 무거움은 해결하지 못한 상태였습니다.

### 📖 개념 가이드 3: 3세대 혁명, RSC (React Server Components)

이제 우리가 배울 Next.js App Router가 제시하는 완전히 새로운 패러다임, **RSC(서버 컴포넌트)** 의 시대가 열렸습니다. RSC는 아주 급진적인 철학을 가졌습니다.

> **"서버에서 실행될 코드는 아예 브라우저로 보내지도 말자!"**

> 💡 **찰떡 비유: 최고급 장비로 시공 완료된 '붙박이 가구'**
>
> 이제는 서버라는 가구 공장에서 전문가들이 최고급 장비(데이터베이스 연결)를 사용해 가구를 벽에 아예 완벽하게 설치해 버립니다. 브라우저(고객)는 망치나 톱 같은 도구(JS 라이브러리)나 설명서를 전달받을 필요가 0%입니다. 그저 집에 들어와 이미 완벽하게 설치된 가구를 즉시 사용하기만 하면 됩니다.

```tsx
/* [Anatomy]: Next.js의 근본, RSC의 압도적인 간결함 */

// 1. 주목! 파일 맨 위에 'use client'라는 글자가 없습니다.
// 즉, 이 코드는 브라우저 밖으로 절대 나가지 않고 오직 서버 안에서만 실행됩니다.
import db from '@/lib/db'; // 그래서 DB에 직접 연결해도 해킹당할 위험이 0%입니다.

// 2. 컴포넌트 함수 자체에 async 하나만 툭 붙여줍니다. (useState, useEffect 완전 증발!)
export default async function ProductPage() {

  // 3. 서버 내부 통신입니다. 인터넷 네트워크를 타지 않으므로 폭포수 지연이 0초에 가깝습니다.
  // 데이터를 즉시 쏙 뽑아옵니다.
  const data = await db.product.findMany();

  // 4. 브라우저는 무거운 로직 다 빼고, 오직 이 '결과물(HTML 껍데기와 글자)'만 배달받습니다!
  return (
    <div>
      {data.map((item) => (
        <div key={item.id}>{item.name}</div>
      ))}
    </div>
  );
}
```

> 🔍 **왜 RSC가 혁명일까요? (Zero Bundle Size)**
>
> 데이터베이스를 조회하는 무거운 로직이나, 어마어마한 용량의 외부 라이브러리를 설치해서 사용하더라도, 브라우저로 전송되는 자바스크립트 용량은 **'0바이트'** 입니다. 브라우저는 오직 서버가 연산해 낸 '결과물'만 낼름 받아먹기 때문에 앱이 깃털처럼 가벼워집니다.

### 🔬 Deep Dive: RSC의 4단계 동작 시뮬레이션

그렇다면 서버에서 브라우저까지 도대체 어떻게 배달되는 걸까요? 딱 4단계만 기억하세요.

1. **요청 (Request):** 사용자가 페이지 주소를 입력하고 접속합니다.
2. **서버 실행 (Server Execution):** Next.js 서버가 컴포넌트를 슥 훑고, 서버 내부에서 데이터베이스를 즉시 뒤져서 데이터를 가져옵니다.
3. **직렬화 (Serialization):** 서버는 이 결과물을 무거운 자바스크립트 코드가 아니라, **'RSC Payload'** 라는 아주 가벼운 텍스트 지시서(JSON)로 압축합니다. (예: "1번 구역에는 '아이폰', 2번 구역에는 '갤럭시' 글자만 넣어라")
4. **전송 및 반영 (Streaming):** 이 가벼운 텍스트 지시서만 인터넷 망을 타고 브라우저로 날아갑니다. 브라우저는 지시서를 보자마자 1초 만에 화면을 완성합니다.

### 🤝 개념 가이드 4: 하이브리드 모델과 `'use client'`의 비밀

여기서 아주 똑똑한 질문이 나올 수 있습니다.

> "그럼 화면에 '좋아요' 버튼을 누르면 숫자가 올라가야 하는데, 렌더링을 서버에서 다 끝내버리면 브라우저에서는 어떻게 클릭 이벤트를 작동시키나요?"

정답입니다. 모든 것을 서버에서만 할 수는 없습니다. 사용자와 상호작용(`onClick` 이벤트, `useState` 상태 변화 등)이 필요한 요소는 브라우저(클라이언트)의 힘을 빌려야 합니다. 그래서 Next.js는 **RSC와 클라이언트 컴포넌트의 하이브리드(Hybrid) 모델**을 채택했습니다.

> 💡 **찰떡 비유: 고급 레스토랑의 주방과 홀 직원**
>
> - **서버 컴포넌트 (주방):** 보이지 않는 곳에서 무거운 장비(DB)로 요리(화면)를 완성합니다. 홀(브라우저)로 절대 나가지 않습니다.
> - **클라이언트 컴포넌트 (홀 직원):** 완성된 요리에 장식을 더하고, 손님(사용자)에게 다가가 직접 주문(클릭 이벤트)을 받고 대화를 나눕니다.

**출입증, `'use client'`**

Next.js는 기본적으로 내가 짠 모든 코드를 주방(서버)에 꽁꽁 가둬둡니다. (보관 원칙) 하지만 "이 버튼은 클릭 기능이 필요해!"라고 판단되면, 해당 컴포넌트 파일 맨 위에 `'use client'`라는 라벨을 적어줍니다. 이것은 주방에서 홀로 나가기 위한 **'출입증'** 입니다.

이 출입증이 붙은 아주 작은 조각(버튼 등)만 브라우저로 전송됩니다. 무거운 짐은 서버에 다 던져두고, 클릭 같은 가벼운 기능만 브라우저로 보내는 최적화. **"보낼 필요 없는 코드는 보내지 않는다"** 는 철학이 바로 Next.js가 미친 듯이 빠른 이유입니다.

### 🚀 강의를 마치며

오늘 우리는 폭포수처럼 쏟아지던 렌더링의 고통(CSR)을 딛고, 코드 용량을 0으로 만들어버리는 서버 컴포넌트(RSC)의 압도적인 기술력을 머릿속에 완벽하게 집어넣었습니다. 무거운 것은 서버에, 가벼운 상호작용은 클라이언트에 맡기는 이 하이브리드 설계는 앞으로 여러분이 작성할 모든 Next.js 코드의 뼈대가 될 것입니다.

하지만 한 가지 거대한 미스터리가 남았습니다. 주방(서버)에서 완성된 정적인 HTML 껍데기가 브라우저에 먼저 도착하고, 그 뒤에 홀 직원(클라이언트 컴포넌트)의 자바스크립트가 도착한다면... 도대체 이 둘은 어느 타이밍에, 어떻게 하나로 찰칵 결합되어 멈춰있던 화면에 생명을 불어넣는 것일까요?

정적인 찰흙 인형에 숨결을 불어넣는 이 마법 같은 과정을 우리는 **'하이드레이션(Hydration)'** 이라고 부릅니다. 다음 시간에는 이 하이드레이션 과정과, 전 세계 개발자의 영혼을 털어버리는 악명 높은 '하이드레이션 미스매치 에러'의 정체를 완벽하게 파헤쳐 보겠습니다.

## 면접 대비 핵심 문제 (Part 1)

### Q1. Library와 Framework의 차이를 설명하고, Next.js가 Framework인 이유를 말해주세요.

<details>
<summary>모범 답안</summary>

**Library**는 개발자가 필요한 기능을 직접 호출하여 사용하는 도구입니다. React가 대표적으로, 라우팅·상태 관리·번들링 등을 개발자가 직접 선택하고 조립해야 합니다.

**Framework**는 이미 전체 구조(뼈대)가 갖춰져 있고, 개발자의 코드가 그 안에 끼워져 호출되는 구조입니다. 이것을 **제어의 역전(Inversion of Control, IoC)** 이라고 합니다.

Next.js가 Framework인 이유는:
- 파일 시스템 기반 라우팅을 자동으로 처리합니다.
- 렌더링 방식(SSR, RSC 등)을 프레임워크가 제어합니다.
- 개발자는 `export default`로 컴포넌트를 넘겨주기만 하면, 언제·어떻게 렌더링할지는 Next.js가 결정합니다.
- **"설정(Configuration)보다 관습(Convention)"** 이라는 철학을 따릅니다.

</details>

### Q2. Next.js의 파일 시스템 라우팅이란 무엇이며, 기존 React Router 방식과 비교해 어떤 장점이 있나요?

<details>
<summary>모범 답안</summary>

**파일 시스템 라우팅**이란 `app` 디렉토리 내 폴더 구조가 곧 URL 경로가 되는 방식입니다. `app/about/page.tsx` 파일을 만들면 자동으로 `/about` 경로가 생성됩니다.

**기존 React Router 방식의 문제점:**
- 별도 설정 파일에서 URL과 컴포넌트를 수동으로 매핑해야 합니다.
- 페이지가 늘어날수록 설정 코드가 비대해집니다.
- 등록 누락이나 오타로 인한 404 에러가 발생하기 쉽습니다.

**파일 시스템 라우팅의 장점:**
- 설정 파일이 불필요합니다.
- 폴더 구조만 보고 사이트 구조를 직관적으로 파악할 수 있습니다.
- `[id]` 같은 동적 세그먼트를 통해 무한한 동적 라우팅을 하나의 파일로 처리할 수 있습니다.

</details>

---

### Q3. CSR, SSR, RSC의 차이를 각각의 동작 방식과 장단점 중심으로 설명해주세요.

<details>
<summary>모범 답안</summary>

| 구분 | 동작 방식 | 장점 | 단점 |
|---|---|---|---|
| **CSR** | 브라우저가 빈 HTML을 받고, JS를 다운로드한 후 클라이언트에서 렌더링 | 인터랙션이 풍부한 SPA 구현에 유리 | 초기 로딩 느림(흰 화면), SEO 불리, Waterfall 현상 |
| **SSR** | 서버가 완성된 HTML을 미리 그려서 전송, 이후 JS로 하이드레이션 | 빠른 초기 화면(FCP), SEO 유리 | 여전히 거대한 JS 번들 전송 필요, 하이드레이션 비용 존재 |
| **RSC** | 서버에서 컴포넌트를 실행하고, 결과만 RSC Payload로 전송. 서버 코드는 브라우저에 전송되지 않음 | **Zero Bundle Size** (JS 전송량 0), DB 직접 접근 가능, 보안 우수 | 상호작용이 필요한 경우 별도 클라이언트 컴포넌트 필요 |

**RSC의 핵심:** 서버에서 실행되는 코드(DB 조회, 외부 API 호출 등)는 브라우저로 전송되지 않으며, 오직 연산 결과(RSC Payload)만 전달됩니다. 이를 통해 번들 크기를 획기적으로 줄일 수 있습니다.

</details>

---

### Q4. RSC Payload란 무엇이며, 서버에서 브라우저까지 전달되는 과정을 설명해주세요.

<details>
<summary>모범 답안</summary>

**RSC Payload**는 서버 컴포넌트의 렌더링 결과물을 직렬화(Serialization)한 가벼운 JSON 형태의 데이터입니다. 무거운 자바스크립트 코드가 아니라, 화면을 구성하기 위한 "지시서" 역할을 합니다.

**전달 과정 4단계:**

1. **요청 (Request):** 사용자가 페이지에 접속합니다.
2. **서버 실행 (Server Execution):** Next.js 서버가 컴포넌트를 실행하고 DB 조회 등을 수행합니다.
3. **직렬화 (Serialization):** 결과물을 RSC Payload라는 경량 텍스트 포맷으로 변환합니다.
4. **스트리밍 (Streaming):** RSC Payload가 브라우저로 전송되고, 브라우저는 이를 기반으로 즉시 화면을 완성합니다.

</details>

### Q5. Next.js App Router에서 `'use client'` 지시어의 역할과, 서버/클라이언트 컴포넌트를 나누는 기준을 설명해주세요.

<details>
<summary>모범 답안</summary>

Next.js App Router에서 **모든 컴포넌트는 기본적으로 서버 컴포넌트**입니다. `'use client'`는 해당 컴포넌트를 클라이언트 컴포넌트로 전환하는 지시어(directive)입니다.

**서버 컴포넌트를 사용해야 하는 경우:**
- DB 조회, 파일 시스템 접근 등 서버 자원이 필요할 때
- 외부 API 호출 등 민감한 키가 포함된 작업
- 상호작용이 없는 정적 UI (텍스트, 이미지 목록 등)

**클라이언트 컴포넌트(`'use client'`)를 사용해야 하는 경우:**
- `onClick`, `onChange` 등 이벤트 핸들러가 필요할 때
- `useState`, `useEffect` 등 React 훅을 사용할 때
- 브라우저 API(`window`, `localStorage` 등)에 접근할 때

**핵심 원칙:** "보낼 필요 없는 코드는 보내지 않는다." 무거운 로직은 서버에, 가벼운 상호작용만 클라이언트에 배치하여 번들 크기를 최소화합니다.

</details>

### Q6. Next.js에서 동적 라우팅(`[id]`)은 어떻게 동작하며, 실제 사용 예시를 설명해주세요.

<details>
<summary>모범 답안</summary>

**동적 라우팅**은 폴더명을 대괄호(`[]`)로 감싸서 정의합니다. 이 폴더는 어떤 값이든 받아들이는 변수 역할을 합니다.

```
app/products/[id]/page.tsx
```

사용자가 `/products/galaxy`에 접속하면:
1. Next.js가 `products` 폴더 하위에서 `galaxy` 폴더를 찾습니다.
2. 정확히 일치하는 폴더가 없으면 `[id]` 동적 세그먼트에 매칭합니다.
3. `"galaxy"` 값이 `params.id`로 전달됩니다.

```tsx
// app/products/[id]/page.tsx
export default async function ProductPage({ params }: { params: { id: string } }) {
  const { id } = await params; // "galaxy"
  const product = await db.product.findUnique({ where: { slug: id } });
  return <h1>{product.name}</h1>;
}
```

이를 통해 상품이 수백만 개여도 하나의 페이지 파일로 모두 처리할 수 있습니다.
