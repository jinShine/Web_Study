## 06강. System Setup & CLI: 1분 만에 끝내는 최신 실무 환경 세팅

### 1. 들어가며: 이론에서 실전으로

지난 시간까지 우리는 Next.js가 가진 철학적 배경과 렌더링의 원리라는 깊은 바다를 유영하며 이론적인 토대를 쌓았습니다. 이제는 그 이론이라는 잠수복을 벗고, 실제로 모래사장 위에서 코드를 쌓아 올릴 시간입니다.

오늘부터 시작되는 Chapter 1에서는 개발 환경 구축부터 라우팅 시스템까지, 내 손으로 직접 코드를 작성하며 Next.js의 근육을 키워보겠습니다. 그 첫 번째 단계로, 오늘은 우리 컴퓨터를 Next.js가 가장 좋아하는 상태로 세팅하고, 공식 CLI 도구인 `create-next-app`을 사용해 단 1분 만에 실무 수준의 프로젝트 환경을 구축합니다.

### 2. Node.js 버전 확인 (Prerequisites)

가장 먼저 체크해야 할 것은 우리 컴퓨터의 심장인 Node.js 버전입니다. Next.js 15 이상을 사용하려면 버전 관리가 매우 중요합니다.

#### 2.1 버전 확인하기

터미널(Terminal) 혹은 명령 프롬프트(CMD)를 열고 아래 명령어를 입력하여 현재 버전을 확인해 보세요.

```bash
node -v
```

#### 2.2 버전 호환성 가이드 (Next.js 15 & 2026년 기준)

| 구분 | 버전 |
|---|---|
| **권장 사항** | v20.x 이상 (Active LTS) |
| **최소 사양** | v18.18.0 이상 |
| **조치 필요** | v18 미만(예: v16, v14)이라면 업데이트 필수 |

> 💡 **Why? 왜 최신 버전을 써야 하나요?**
>
> 기업 실무에서는 LTS(Long Term Support) 버전이 '안정성'의 척도입니다. 특히 Next.js 15는 React 19를 기반으로 동작하는데, React 19의 최신 서버 컴포넌트(RSC) API와 빌드 최적화 기능들을 100% 활용하려면 구버전 Node.js는 적합하지 않습니다. 따라서 우리는 가장 안정적이고 강력한 v20 이상의 LTS 버전으로 통일하고 진행하겠습니다.

### 3. 프로젝트 생성 (Create Project)

Node.js 버전이 준비되었다면 이제 곧바로 프로젝트를 생성해 보겠습니다. 우리는 수동 설치 대신 Next.js 팀이 공식적으로 제공하고 유지 보수하는 가장 강력한 자동화 도구인 `create-next-app`을 사용할 것입니다.

#### 3.1 설치 명령어 실행

프로젝트를 생성하고 싶은 폴더 경로로 이동한 뒤, 다음 명령어를 입력하세요.

```bash
npx create-next-app@latest
```

- `npx`: 내 컴퓨터에 설치 도구를 영구 저장하지 않고, 항상 최신 버전을 실행하게 해주는 고마운 도구입니다.
- `@latest`: 현재 시점(2026년)에서 가장 최신 버전인 Next.js 15 이상의 설치를 보장합니다.

### 4. 상세 설정 마법사 (Configuration Wizard)

명령어를 실행하면 설치 마법사가 실행되며 여러분에게 프로젝트의 아키텍처를 결정하는 몇 가지 중요한 질문을 던집니다.

#### 4.1 프로젝트 이름 (Project Name)

```
What is your project named? … my-next-app
```

- 영문으로 원하는 이름을 입력합니다. (예: `my-next-app`)
- 보통 소문자와 하이픈(-)을 사용하는 케밥 케이스(kebab-case)가 관례입니다.

#### 4.2 TypeScript 사용 (Type Safety)

```
Would you like to use TypeScript? … Yes
```

- 선택: **Yes (필수)**
- 2026년 현재 웹 개발 환경에서 타입 안정성은 타협할 수 없는 필수 요소입니다. 컴파일 단계에서 오류를 잡아내어 런타임 에러를 획기적으로 줄여줍니다.

#### 4.3 ESLint 사용 (Linting)

```
Would you like to use ESLint? … Yes
```

- 선택: **Yes**
- 코드의 문법적 오류나 스타일을 잡아주는 최소한의 안전장치입니다.

#### 4.4 Tailwind CSS 사용 (Styling)

```
Would you like to use Tailwind CSS? … Yes
```

- 선택: **Yes**
- Next.js는 공식적으로 Tailwind CSS를 기본 스타일링 엔진으로 밀고 있습니다. `postcss.config.js` 및 `tailwind.config.ts` 파일이 자동으로 생성되어 즉시 유틸리티 클래스를 사용할 수 있습니다.

#### 4.5 `src` 디렉토리 사용 (Project Structure)

```
Would you like your code inside a src/ directory? … Yes
```

- 선택: **Yes (강력 권장)**
- 프로젝트 루트에는 설정 파일만 두고, 실제 소스 코드는 `src` 폴더 안에 격리하여 관리하는 것이 유지 보수성을 높이는 실무 표준입니다.

#### 4.6 App Router 사용 (Routing)

```
Would you like to use App Router? … Yes
```

- 선택: **Yes (매우 중요)**
- 이것이 우리가 배울 핵심입니다. 실수로 `No`를 선택하면 구버전(Pages Router)이 설치되니 주의하세요! App Router는 서버 컴포넌트, 스트리밍, 레이아웃 시스템을 완벽하게 지원합니다.

#### 4.7 Turbopack 사용 (Dev Server Engine) [Next.js 15 핵심]

```
Would you like to use Turbopack for next dev? … Yes
```

- 선택: **Yes (필수)**

> 💡 **기술적 배경 (Deep Dive)**
>
> 과거 Webpack은 무거워서 프로젝트가 커질수록 서버 켜지는 시간이 느렸습니다. 하지만 Next.js 15부터는 Rust로 작성된 초고속 번들러인 **Turbopack**이 개발 모드(dev)에서 안정화(Stable)되었습니다. 이걸 켜면 서버 시작 속도가 최대 700배 빨라집니다.

#### 4.8 Import Alias 커스터마이징

```
Would you like to customize the default import alias (@/*)? … No
```

- 선택: **No (기본값 사용)**
- 복잡한 상대 경로(`../../components`) 대신, `@/components`처럼 깔끔하게 파일을 불러올 수 있습니다.

### 5. 개발 서버 실행 (Run Dev Server)

이 모든 선택을 마치면 `create-next-app`은 우리가 선택한 옵션에 맞춰 최적의 보일러플레이트 코드를 생성해 줍니다. 과거 Webpack 설정으로 며칠 밤을 새우던 작업이 단 1분 만에 끝난 것입니다.

#### 5.1 프로젝트 폴더로 이동 및 실행

```bash
cd my-next-app
npm run dev
```

- `cd my-next-app`: 방금 생성한 프로젝트 디렉토리로 이동합니다.
- `npm run dev`: `package.json`에 정의된 스크립트를 실행하여 로컬 개발 서버를 띄웁니다. 방금 우리가 Turbopack을 선택했기 때문에 서버가 순식간에 켜질 겁니다.

#### 5.2 실행 확인

터미널에 Ready 문구와 함께 로컬 주소가 나타납니다.

```
- Ready in 1234ms (Turbopack)
- Local: http://localhost:3000
```

#### 5.3 브라우저 접속

이제 웹 브라우저를 열고 주소창에 `http://localhost:3000`을 입력해 보세요.

> **결과:** 화면 중앙에 Next.js 로고가 박힌 검은색 환영 페이지가 보인다면 여러분은 성공적으로 Next.js라는 우주선에 탑승하신 것입니다.

## 07강. Project Structure: 파일 시스템 라우팅의 절대 규칙과 위치 병합(Colocation)

### 학습 목표

- Next.js 프로젝트의 루트 디렉토리 구조와 설정 파일들의 역할 이해
- App Router(`src/app`)의 핵심 철학: 파일 시스템 기반 라우팅
- 필수 파일(`page.tsx`, `layout.tsx`)의 엄격한 명명 규칙
- Colocation(위치 병합) 패턴을 통한 안전한 컴포넌트 관리

### 1. 프로젝트 구조 한눈에 보기 (Overview)

지난 시간 CLI를 통해 생성한 프로젝트를 VS Code로 열어볼 시간입니다. 처음 마주하는 이 구조는 단순한 파일 배치가 아니라, Next.js가 제시하는 "반드시 이렇게 개발해야 한다"는 엄격한 법전입니다.

```
my-next-app/
├── node_modules/          # [시스템] 외부 라이브러리 저장소 (절대 수정 금지)
├── public/                # [자산] 정적 파일 보관소 (이미지, 폰트, 로봇.txt)
├── src/
│   └── app/               # [핵심] App Router의 심장부 (실제 코딩 위치)
├── eslint.config.mjs      # [규칙] 코드 품질 검사 설정 (Flat Config)
├── next.config.ts         # [설정] Next.js 프레임워크 전용 설정
├── tailwind.config.ts     # [스타일] CSS 프레임워크 설정
├── tsconfig.json          # [언어] TypeScript 컴파일 규칙
└── package.json           # [신분증] 프로젝트 정보 및 의존성 명세서
```

### 2. 루트 설정 파일: 프로젝트의 뼈대

프로젝트의 뿌리(Root)에 위치한 설정 파일들은 앱의 규칙을 결정짓는 중요한 문서들입니다.

| 파일 | 별명 | 역할 |
|---|---|---|
| **`package.json`** | 주민등록증 | 프로젝트의 이름, 버전, 의존 라이브러리 목록, `npm run dev` 같은 명령어(Script) 정의 |
| **`tsconfig.json`** | 엄격한 감독관 | TypeScript 컴파일러 설정. Next.js가 자동 구성하므로 함부로 수정/삭제 시 전체 에러 발생 가능 |
| **`tailwind.config.ts`** | 디자인 시스템 설정집 | 색상 팔레트, 폰트 크기, 여백 등 Tailwind CSS 스타일링 규칙 정의 |
| **`eslint.config.mjs`** | 코드 경찰관 | 세미콜론 누락, 미사용 변수 등을 감시. Next.js 15부터 최신 Flat Config 형식 사용 |
| **`next.config.ts`** | 중앙 관제탑 | Next.js 서버 동작 제어. 이미지 최적화 도메인, 리다이렉트 규칙 등 담당 |

### 3. App Router의 심장: `src/app`

`src` 폴더는 우리가 작성할 모든 소스 코드가 격리되어 보호되는 공간입니다. 그중 `app` 디렉토리는 **라우터 그 자체**입니다.

#### 절대 규칙 1: `page.tsx` (가게 간판)

Next.js는 오직 약속된 이름을 가진 파일에게만 특별한 권한을 부여합니다. `page.tsx`는 해당 경로의 유일한 UI를 담당합니다. 파일명이 `Page.tsx`나 `index.tsx`여서는 안 되며, **반드시 소문자 `page.tsx`** 여야만 브라우저에서 화면이 렌더링됩니다.

#### 절대 규칙 2: `layout.tsx` (인테리어 골조)

여러 페이지가 공유하는 공통의 UI(껍데기)를 정의합니다. 페이지 이동 시에도 다시 생성되지 않고 **상태가 유지(Persist)** 됩니다. SEO를 위한 메타데이터(`title`, `description`)도 주로 여기서 설정합니다.

#### 실습 코드: Root Layout 분석

`app/layout.tsx` 파일을 열어보면 다음과 같은 구조를 가집니다.

```tsx
// app/layout.tsx
// 이 파일은 애플리케이션의 최상위 레이아웃(Root Layout)입니다.

import type { Metadata } from "next";
import "./globals.css";

export const metadata: Metadata = {
  title: "Create Next App",
  description: "Generated by create next app",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    // ⚠️ 주의: html과 body 태그는 오직 'Root Layout'에만 존재해야 합니다!
    <html lang="en">
      <body>
        {/* 1. 네비게이션 바: 여기에 배치하면 모든 페이지에서 유지됩니다. */}
        <nav className="p-4 bg-gray-100">
            My Navbar (공통 영역)
        </nav>

        {/* 2. Main Content: 우리가 만든 page.tsx가 이 'children' 자리에 렌더링됩니다. */}
        <main>{children}</main>

        {/* 3. 푸터: 여기에 배치하면 모든 페이지 하단에 고정됩니다. */}
        <footer className="p-4 bg-gray-800 text-white">
            My Footer (공통 영역)
        </footer>
      </body>
    </html>
  );
}
```

> 🔍 **핵심 포인트**
>
> - **`children` Prop:** 하위 경로의 `page.tsx`나 중첩된 `layout.tsx` 컴포넌트가 들어오는 슬롯(구멍)입니다.
> - **`html`, `body` 태그:** Root Layout은 반드시 `html`과 `body` 태그를 포함해야 합니다. (반대로, 나중에 만들 하위 레이아웃에는 **절대 넣으면 안 됩니다!**)

#### 상황별 예약 파일 (Special Files)

Next.js는 상황에 따른 UI도 파일 이름으로 약속해 두었습니다.

| 파일 | 역할 |
|---|---|
| **`loading.tsx`** | 데이터 로딩 중 보여줄 UI (React Suspense 자동 적용) |
| **`error.tsx`** | 에러 발생 시 보여줄 UI (Error Boundary 자동 적용) |
| **`not-found.tsx`** | 존재하지 않는 경로 접근 시 보여줄 404 UI |

### 4. Colocation (위치 병합): 혁신적인 파일 관리

Next.js App Router의 가장 강력한 철학인 **Colocation(위치 병합)** 에 대해 알아봅니다. 이는 "관련된 코드는 멀리 두지 말고 한 방(폴더)에 같이 두자"는 원칙입니다.

#### 과거의 문제점 (Pages Router)

과거에는 폴더 안에 파일을 만드는 순간, 그것이 곧바로 인터넷 주소(URL)가 되었습니다.

- `dashboard/Chart.tsx` 생성 → `site.com/dashboard/Chart`로 접근 가능
- **문제:** 부품으로 쓰려던 컴포넌트나 스타일 파일이 의도치 않게 페이지로 노출되는 보안/구조적 결함 발생

#### 현재의 혁신 (App Router)

지금은 폴더 안에 파일이 있어도, **이름표가 `page.tsx`가 아니면 절대 라우트로 인식되지 않습니다.**

```
src/app/
└── dashboard/
    ├── page.tsx      // (O) 브라우저 주소: /dashboard (유일한 진입점)
    ├── layout.tsx    // (O) /dashboard 하위 페이지의 레이아웃 (약속된 파일)
    ├── Chart.tsx     // (X) 라우트 절대 아님 (단순 컴포넌트)
    └── styles.css    // (X) 라우트 절대 아님 (해당 페이지 전용 스타일)
```

> 🔍 **해석**
>
> - 사용자가 `/dashboard`에 접속하면 → `page.tsx`라는 문이 있으므로 열립니다.
> - 사용자가 `/dashboard/Chart`에 접속하면 → `Chart.tsx`는 문(page)이 아니므로 **404 Error**가 발생합니다.

이 철칙 덕분에 우리는 더 이상 컴포넌트를 숨기기 위해 머나먼 `components` 폴더를 찾아 헤맬 필요가 없습니다. `page.tsx`는 손님에게 보여줄 가게 간판이고, 나머지는 직원 전용 물품으로서 같은 방에서 안전하게 관리될 수 있습니다.

### 5. `public` 폴더: 정적 자산의 창고

마지막으로 루트에 있는 `public` 폴더를 살펴봅니다. 이곳은 소스 코드가 아닌, 외부로 공개된 **정적 자산(Static Assets)** 을 보관하는 창고입니다.

- **용도:** 이미지(`.png`, `.jpg`), 폰트, `robots.txt`, `favicon.ico` 등
- **접근 방식:** `public` 폴더 내에 `logo.png`를 넣었다면, 브라우저 주소창이나 코드에서 `/logo.png`로 바로 접근합니다.

```tsx
// 예시: public/logo.png 파일을 사용하는 코드
import Image from 'next/image';

export default function Page() {
  return (
    // 별도의 import 없이 절대 경로 '/'로 바로 접근
    // 여기서 '/'는 public 폴더를 의미합니다.
    <Image src="/logo.png" alt="Logo" width={100} height={100} />
  );
}
```

### 🧪 테스트 가이드 (제대로 이해했는지 확인하기)

1. **서버 구동 확인:** 터미널에서 `npm run dev`를 실행했을 때 에러 없이 `localhost:3000`이 열리는지 확인합니다.
2. **파일명 강제성 확인:** `src/app/page.tsx` 파일명을 `home.tsx`로 변경해 보세요. 브라우저에 404 에러가 뜨는지 확인한 후 다시 원복합니다.
3. **루트 레이아웃 수정:** `layout.tsx`의 네비게이션 문구를 수정해 보세요. 브라우저에서 모든 페이지 상단에 수정한 문구가 뜨는지 확인합니다.
4. **정적 자산 접근:** `public/favicon.ico` 파일을 브라우저 주소창에 `localhost:3000/favicon.ico`로 입력했을 때 이미지가 잘 뜨는지 확인합니다.
5. **컴포넌트 병합 테스트:** `src/app` 폴더 안에 `MyComponent.tsx`를 만들고 `page.tsx`에서 불러와 사용해 보세요. `/MyComponent` 주소로 접속했을 때 404 에러가 뜨면 정상입니다.

### 📌 핵심 요약 (Core Summary)

- **명명 규칙 준수:** `page.tsx`와 `layout.tsx`는 대소문자까지 정확히 지켜야 하는 시스템 파일입니다.
- **Root Layout의 책임:** 최상위 레이아웃은 반드시 `<html>`과 `<body>` 태그를 포함해야 합니다.
- **Colocation의 이점:** 부품 컴포넌트를 `page.tsx`와 같은 폴더에 두어 관리 편의성을 극대화할 수 있습니다.
- **상태 유지:** 레이아웃은 페이지 이동 시에도 언마운트되지 않아 검색어 입력값이나 스크롤 위치 등을 보존하기 유리합니다.
- **Public의 역할:** `/` 경로로 시작하는 모든 정적 파일(이미지 등)은 `public` 폴더를 참조합니다.

## 08강. Pages & Layouts: 내 손으로 직접 뚫어보는 첫 라우트와 중첩(Nested) 레이아웃의 미학

### 학습 목표

- 파일 시스템 라우팅(File-system Routing)의 개념을 이해하고 직접 폴더를 통해 주소를 생성합니다.
- `page.tsx`와 `layout.tsx`의 유기적인 결합 방식을 익힙니다.
- 중첩 레이아웃(Nested Layouts)이 어떻게 UI 위계를 형성하는지 체득합니다.

### 1. 첫 번째 페이지 만들기: `/about`

Next.js에서는 컴포넌트를 만들고 라우터 설정 파일을 열어 경로를 입력하는 복잡한 과정이 필요 없습니다. **'올바른 위치에 폴더와 파일을 생성하는 것'** 하나면 충분합니다.

#### 🛠️ 실습 단계

1. `src/app` 폴더 아래에 `about`이라는 이름의 새로운 폴더를 생성합니다.
2. 생성한 폴더 안에 `page.tsx` 파일을 만들고 아래 코드를 작성합니다.

```tsx
// src/app/about/page.tsx

export default function AboutPage() {
  return (
    <div className="p-10">
      <h1 className="text-3xl font-bold">안녕하세요, About 페이지입니다!</h1>
      <p className="mt-4 text-gray-600">
        우리가 직접 만든 폴더가 곧바로 URL 주소가 되었습니다.
        이것이 바로 파일 시스템 라우팅(File-system Routing)의 마법입니다.
      </p>
    </div>
  );
}
```

> 💡 **코드 상세 설명**
>
> - **경로와 URL의 매핑:** 파일이 `src/app/about`에 위치하므로, 브라우저 주소는 자동으로 `/about`이 됩니다.
> - **파일명 규칙:** 반드시 소문자 `page.tsx`여야 합니다. `About.tsx`나 `Page.tsx`는 Next.js가 인식하지 못해 404 에러를 발생시킵니다.
> - **Export 규칙:** 함수 이름(예: `AboutPage`)은 자유롭게 지을 수 있지만, 반드시 `export default`를 사용해야 Next.js가 페이지로 렌더링할 수 있습니다.
> - **Fast Refresh:** 코드를 저장하면 서버를 재시작할 필요 없이 즉시 브라우저에 반영됩니다.

### 2. 중첩 레이아웃(Nested Layout)의 적용

특정 섹션에서만 공통으로 유지되어야 하는 디자인(네비게이션, 사이드바 등)이 있다면 해당 폴더에 `layout.tsx`를 추가하면 됩니다.

#### 🛠️ 실습 단계

1. `src/app/about` 폴더 안에 `layout.tsx` 파일을 생성합니다.
2. 아래 코드를 작성하여 About 섹션만의 '껍데기'를 만듭니다.

```tsx
// src/app/about/layout.tsx

export default function AboutLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <section className="border-4 border-blue-500 m-4 p-4 rounded-lg">
      <nav className="bg-blue-100 p-4 mb-4 rounded">
        <h2 className="text-xl font-bold text-blue-800">
          🏢 About 섹션 전용 네비게이션
        </h2>
        <ul className="flex gap-4 mt-2">
            <li>소개</li>
            <li>연혁</li>
            <li>팀원</li>
        </ul>
      </nav>

      {/* page.tsx의 내용이 들어가는 자리 */}
      <main className="bg-white">
        {children}
      </main>
    </section>
  );
}
```

> 💡 **코드 상세 설명**
>
> - **`children` Props:** 레이아웃 컴포넌트는 반드시 `children`을 받아야 합니다. 이 자리에 하위 `page.tsx` 내용이 자동으로 주입됩니다.
> - **위계 구조:** `layout.tsx`는 **액자**이고, `page.tsx`는 그 안에 끼워지는 **사진**입니다. 이제 `/about` 경로로 접속하면 파란색 테두리와 전용 네비게이션이 포함된 화면을 보게 됩니다.

### 3. 레이아웃 중첩의 미학: `/about/company`

Next.js 라우팅의 진가는 폴더가 깊어질 때 나타납니다. 레이아웃은 폭포수처럼 상위에서 하위로 물려 내려갑니다.

#### 🛠️ 실습 단계

1. `src/app/about` 폴더 안에 다시 `company` 폴더를 만듭니다.
2. `company` 폴더 안에 `page.tsx`를 만들고 아래 내용을 작성합니다.

```tsx
// src/app/about/company/page.tsx

export default function CompanyPage() {
  return (
    <div>
      <h2 className="text-2xl font-bold text-red-600">우리 회사를 소개합니다.</h2>
      <p className="text-black">이곳은 /about/company 경로입니다.</p>
    </div>
  );
}
```

#### 🥪 결과 분석: 샌드위치 조립 방식

브라우저에서 `localhost:3000/about/company`에 접속하면 Next.js는 아래 순서대로 화면을 조립합니다.

1. **1단계:** 가장 바깥쪽의 Root Layout (`src/app/layout.tsx`)을 깝니다.
2. **2단계:** 그 위에 About Layout (`src/app/about/layout.tsx`)을 얹습니다.
3. **3단계:** 마지막으로 가장 안쪽에 Company Page 내용을 채웁니다.

### 🧪 테스트 가이드

1. **기본 라우팅 확인:** 브라우저 주소창에 `localhost:3000/about`을 입력했을 때 우리가 만든 `AboutPage` 내용이 뜨나요?
2. **파일명 대소문자 체크:** `page.tsx` 이름을 `Page.tsx`로 바꿔보세요. 404 에러가 뜨는 것을 확인했다면 다시 소문자로 돌려놓으세요.
3. **레이아웃 적용 확인:** `/about` 페이지에 파란색 테두리와 "About 섹션 전용 네비게이션"이 보이나요?
4. **중첩 라우팅 확인:** `localhost:3000/about/company`에 접속해 보세요. 파란색 테두리(About Layout)는 유지된 채 알맹이만 "우리 회사를 소개합니다"로 바뀌었나요?
5. **루트 레이아웃 결합:** 브라우저 탭의 제목(Title)이나 전체 배경 등 루트 레이아웃(`src/app/layout.tsx`)의 설정이 `/about/company`에서도 여전히 유효한가요?

### 📌 핵심 요약 (Cheat Sheet)

- **파일 시스템 라우팅:** 폴더 구조가 곧 URL 주소가 됩니다. 별도의 설정 파일 없이 직관적으로 경로를 설계합니다.
- **`page.tsx` (필수):** 해당 경로의 실제 콘텐츠입니다. 이 파일이 없으면 그 주소는 유효하지 않은 경로(404)가 됩니다.
- **`layout.tsx` (선택):** 해당 폴더와 그 하위의 모든 페이지를 감싸는 공통 UI입니다. 중첩될수록 UI 계층 구조가 형성됩니다.
- **상태 유지 (Persistence):** 페이지가 바뀌어도 상위 레이아웃은 다시 그려지지(Re-render) 않고 상태를 유지합니다.
- **Naming Convention:** 폴더명은 URL 경로가 되고, 파일명은 반드시 `page.tsx`, `layout.tsx`와 같이 Next.js가 정한 약속을 따라야 합니다.

## 09강. Layout Deep Dive: 페이지 이동에도 상태가 유지되는 대시보드 구현

지난 시간 우리는 `layout.tsx`가 하위 페이지들을 감싸는 액자 역할을 한다는 사실을 배웠습니다. 하지만 단순히 정적인 페이지 몇 개를 감싸는 것만으로는 Next.js 레이아웃 시스템의 진짜 가치를 10%도 경험하지 못한 것입니다.

오늘은 실무적이고 강력한 예제인 **관리자 대시보드**를 직접 만들어보며, "페이지가 바뀌어도 레이아웃은 절대 다시 그려지지 않는다"는 불변의 법칙을 확인해 보겠습니다.

### 1. 레이아웃의 불변성: 왜 중요한가?

| 방식 | 동작 |
|---|---|
| **기존 방식** | 메뉴 클릭 시 화면 전체가 하얗게 깜빡이며 새로고침. 사이드바 검색어, 스크롤 위치 등 모든 정보가 초기화 |
| **Next.js 방식** | 레이아웃 컴포넌트는 최초에 딱 한 번만 렌더링. 이후 고정된 채 오직 `page.tsx`만 갈아 끼워짐 |

마치 TV의 테두리는 그대로 있고 화면 속 영상만 바뀌는 것과 같습니다.

### 2. [실습 1] 대시보드 레이아웃(Layout) 설계

먼저 `src/app/dashboard` 폴더를 만들고, 그 안에 `layout.tsx`를 생성합니다.

```tsx
// src/app/dashboard/layout.tsx
import Link from "next/link";

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    // 전체 화면(h-screen)을 flex 컨테이너로 만듭니다.
    <div className="flex h-screen text-white">

      {/* 왼쪽 사이드바 (고정 영역) */}
      <aside className="w-64 bg-gray-900 p-6 flex flex-col border-r border-gray-800">

        {/* 로고 영역 */}
        <h1 className="text-2xl font-bold mb-6 text-emerald-400">
           Dev Dashboard
        </h1>

        {/* ★ 핵심 테스트용 검색창 ★ */}
        <div className="mb-6">
            <input
                type="text"
                placeholder="검색어를 입력해보세요..."
                className="w-full p-2 rounded bg-gray-800 text-sm focus:outline-none focus:ring-2 focus:ring-emerald-500 text-white"
            />
            <p className="text-xs text-gray-500 mt-2">
                ※ 여기에 글을 쓰고 메뉴를 이동해보세요!
            </p>
        </div>

        {/* 네비게이션 메뉴 */}
        <nav className="space-y-2 flex-1 overflow-y-auto">
          <Link href="/dashboard" className="block p-3 rounded hover:bg-gray-800 transition-colors">
            📊 홈 (Overview)
          </Link>
          <Link href="/dashboard/settings" className="block p-3 rounded hover:bg-gray-800 transition-colors">
            ⚙️ 설정 (Settings)
          </Link>
        </nav>

        {/* 하단 나가기 버튼 */}
        <Link href="/" className="block p-3 rounded hover:bg-red-900/30 text-red-400 text-sm mt-4">
            ⬅️ 메인으로 나가기
        </Link>
      </aside>

      {/* 오른쪽 콘텐츠 영역 (가변 영역) */}
      <main className="flex-1 bg-gray-100 text-black p-8 overflow-y-auto">
        {children}
      </main>
    </div>
  );
}
```

> 💡 **코드 상세 설명**
>
> - **`children` Props:** 대시보드 하위의 모든 페이지(`page.tsx`)가 렌더링되어 들어올 구멍(Slot)입니다.
> - **`flex h-screen`:** 전체 화면 높이를 꽉 채우고, 왼쪽(사이드바)과 오른쪽(본문)을 가로로 배치합니다.
> - **`aside` (사이드바):** `w-64`로 너비를 고정했습니다. 이 영역 안에 있는 `input` 태그는 페이지 이동 시 데이터가 날아가는지 확인하는 테스트 장치입니다.
> - **`overflow-y-auto`:** 오른쪽 본문 영역(`main`)에 이 속성을 주어, 본문 내용이 길어져도 사이드바는 움직이지 않고 본문만 따로 스크롤되도록 설정했습니다.
> - **`Link` 컴포넌트:** `<a>` 태그 대신 사용하여 브라우저 새로고침 없이 부분 렌더링이 일어나게 합니다.

>
> - **`h-[1500px]`:** 페이지 길이를 강제로 늘려 스크롤을 발생시킵니다. 본문을 내려도 왼쪽 사이드바는 고정되어 있음을 확인하기 위함입니다.
> - **`grid grid-cols-1 md:grid-cols-3`:** Tailwind의 반응형 그리드입니다. 모바일에서는 1줄, 데스크톱에서는 3줄로 통계 카드를 배치합니다.
> - **`rounded-xl shadow-lg`:** 흰색 배경의 박스에 둥근 모서리와 그림자를 주어 대시보드 본문이 레이아웃 위에 떠 있는 듯한 느낌을 줍니다.

### 4. [실습 3] 설정 페이지 구현

비교를 위해 `src/app/dashboard/settings/page.tsx` 경로에 설정 페이지를 추가합니다.

```tsx
// src/app/dashboard/settings/page.tsx
export default function SettingsPage() {
  return (
    <div className="bg-white p-8 rounded-xl shadow-lg border-l-8 border-emerald-500">
      <h2 className="text-3xl font-bold mb-4">⚙️ 환경 설정</h2>
      <p className="text-gray-600 mb-8">
        여기는 설정 페이지입니다. <br/>
        레이아웃은 그대로 유지된 채, 오직 이 흰색 영역만 교체되었습니다.
      </p>

      <div className="space-y-4 max-w-md">
        <label className="flex items-center space-x-3 p-4 border rounded cursor-pointer hover:bg-gray-50">
           <input type="checkbox" className="w-5 h-5 accent-emerald-600" defaultChecked />
           <span>이메일 알림 받기</span>
        </label>
        <label className="flex items-center space-x-3 p-4 border rounded cursor-pointer hover:bg-gray-50">
           <input type="checkbox" className="w-5 h-5 accent-emerald-600" />
           <span>다크 모드 사용</span>
        </label>
      </div>
    </div>
  );
}
```

> 💡 **코드 상세 설명**
>
> - **`border-l-8 border-emerald-500`:** 왼쪽 테두리에 굵은 초록색 포인트를 주어 메인 페이지와 시각적으로 확연히 다름을 표시했습니다.
> - **checkbox UI:** 설정 페이지의 전형적인 요소들을 배치했습니다. 메인 페이지에서 설정 페이지로 이동할 때, 레이아웃(사이드바)의 검색어가 지워지지 않는다면 성공입니다.

### 🧪 테스트 가이드: 직접 확인해보기

1. **초기 접속:** 브라우저에서 `localhost:3000/dashboard`에 접속합니다.
2. **데이터 입력:** 왼쪽 사이드바의 검색창에 "Next.js 상태 유지 테스트"라고 입력합니다.
3. **페이지 이동:** 사이드바 메뉴에서 '⚙️ 설정 (Settings)' 링크를 클릭합니다.
4. **상태 확인:** 우측 본문 영역이 설정 페이지로 바뀌었음에도, 왼쪽 검색창의 글자가 그대로 유지되어 있는지 확인합니다.
5. **스크롤 테스트:** '홈(Overview)'으로 돌아와 본문 영역을 아래로 끝까지 스크롤 해보세요. 왼쪽 사이드바는 화면에 고정되어 움직이지 않아야 합니다.

### 📌 핵심 요약 (Core Summary)

- **레이아웃의 상태 유지 (Preserving State):** 페이지 전환 시 `layout.tsx`는 언마운트(Unmount)되지 않습니다. 덕분에 입력 폼의 텍스트, 스크롤 위치, 비디오 재생 상태 등이 그대로 보존됩니다.
- **부분 렌더링 (Partial Rendering):** Next.js는 변경된 하위 `children` 페이지만 서버에서 받아와 교체합니다. 전체 페이지를 다시 불러오지 않으므로 깜빡임이 없고 속도가 매우 빠릅니다.
- **클라이언트 사이드 내비게이션:** `Link` 컴포넌트를 사용하면 브라우저의 기본 이동 방식이 아닌, 앱과 같은 부드러운 화면 전환 UX를 제공할 수 있습니다.
- **독립적 스크롤 제어:** 레이아웃 구조 내에서 `overflow-y-auto`를 적절히 배치하면 사이드바는 고정하고 본문만 스크롤 하는 실무적인 대시보드 레이아웃을 쉽게 만들 수 있습니다.

### 🚀 마무리

우리는 정적인 페이지 구조를 완벽하게 장악했습니다. 하지만 쇼핑몰처럼 상품이 10만 개라면 폴더를 10만 개 만들어야 할까요? 그럴 순 없겠죠.

다음 강의에서는 단 하나의 파일로 무한대의 페이지를 생성해내는 Next.js의 진정한 무기, **'Dynamic Routes(동적 라우팅)'** 의 세계로 떠나보겠습니다.

### Q1. Next.js 프로젝트에서 `page.tsx`와 `layout.tsx`의 역할과 차이를 설명해주세요.

<details>
<summary>모범 답안</summary>

**`page.tsx`:**
- 해당 라우트 경로의 **고유한 UI(콘텐츠)**를 담당합니다.
- 반드시 소문자 `page.tsx`여야 하며, `export default`로 내보내야 합니다.
- 이 파일이 존재해야만 해당 폴더가 유효한 URL 경로로 인식됩니다.

**`layout.tsx`:**
- 해당 폴더와 그 하위 모든 페이지를 감싸는 **공통 UI 껍데기**입니다.
- `children` prop을 받아 하위 `page.tsx`를 렌더링하는 슬롯을 제공합니다.
- 페이지 이동 시에도 **언마운트되지 않고 상태가 유지**됩니다 (입력값, 스크롤 위치 등 보존).

**Root Layout의 특수성:**
- `src/app/layout.tsx`는 반드시 `<html>`과 `<body>` 태그를 포함해야 합니다.
- 하위 레이아웃에는 이 태그들을 포함하면 안 됩니다.

</details>

### Q2. Next.js App Router의 Colocation(위치 병합) 패턴이란 무엇이며, 이전 Pages Router와 비교해 어떤 장점이 있나요?

<details>
<summary>모범 답안</summary>

**Colocation(위치 병합)** 이란 관련된 코드(컴포넌트, 스타일, 유틸 등)를 해당 라우트 폴더 안에 함께 두는 패턴입니다.

**과거 Pages Router의 문제:**
- 폴더 안에 파일을 만드는 순간 자동으로 URL이 생성되었습니다.
- `dashboard/Chart.tsx`가 `/dashboard/Chart`로 접근 가능해져 의도치 않은 페이지 노출 발생.
- 컴포넌트를 별도의 `components/` 폴더에 분리해서 관리해야 했습니다.

**App Router의 혁신:**
- **오직 `page.tsx`라는 이름의 파일만** 라우트로 인식됩니다.
- 같은 폴더에 `Chart.tsx`, `styles.css` 등을 두어도 URL로 노출되지 않습니다.
- 관련 코드가 가까이 위치하여 **유지보수성과 가독성**이 크게 향상됩니다.

</details>

### Q3. Next.js에서 중첩 레이아웃(Nested Layouts)이 동작하는 방식과, 이것이 UX에 미치는 영향을 설명해주세요.

<details>
<summary>모범 답안</summary>

**중첩 레이아웃의 동작 방식:**

`/about/company` 경로에 접속하면 Next.js는 다음 순서로 화면을 조립합니다:

1. **Root Layout** (`app/layout.tsx`) - 가장 바깥 껍데기 (`<html>`, `<body>`)
2. **About Layout** (`app/about/layout.tsx`) - About 섹션 전용 UI
3. **Company Page** (`app/about/company/page.tsx`) - 실제 콘텐츠

각 레이아웃은 `children` prop을 통해 하위 레이아웃이나 페이지를 받아 렌더링합니다.

**UX에 미치는 영향:**

- **상태 유지(Preserving State):** 상위 레이아웃은 하위 페이지가 바뀌어도 언마운트되지 않으므로, 검색창 입력값, 스크롤 위치, 사이드바 상태 등이 보존됩니다.
- **부분 렌더링(Partial Rendering):** 변경된 `page.tsx`만 교체하므로 깜빡임 없는 부드러운 전환이 가능합니다.
- **공통 UI 재사용:** 헤더, 사이드바, 푸터 등을 레이아웃 계층에 따라 효율적으로 관리할 수 있습니다.

</details>

### Q4. `<Link>` 컴포넌트와 `<a>` 태그의 차이를 설명하고, 왜 Next.js에서는 `<Link>`를 사용해야 하나요?

<details>
<summary>모범 답안</summary>

| 구분 | `<a>` 태그 | `<Link>` 컴포넌트 |
|---|---|---|
| **페이지 이동 방식** | 브라우저 전체 새로고침 (Hard Navigation) | 클라이언트 사이드 라우팅 (Soft Navigation) |
| **상태 보존** | 모든 클라이언트 상태 소멸 | 레이아웃 상태 유지 |
| **깜빡임** | 하얀 화면 깜빡임 발생 | 깜빡임 없이 부드러운 전환 |
| **Prefetching** | 없음 | 뷰포트에 들어온 링크를 자동으로 미리 로드 |
| **전송 데이터** | 전체 HTML 문서를 새로 요청 | 변경된 RSC Payload만 요청 |

**`<Link>`를 사용해야 하는 이유:**
- `layout.tsx`의 상태 보존(검색어, 스크롤 등)이 가능합니다.
- 부분 렌더링으로 불필요한 네트워크 요청과 렌더링을 방지합니다.
- Prefetching으로 체감 로딩 시간을 0에 가깝게 만듭니다.
- 앱과 같은 부드러운 사용자 경험을 제공합니다.

</details>

### Q5. Next.js App Router의 Special Files(`loading.tsx`, `error.tsx`, `not-found.tsx`)의 역할을 각각 설명해주세요.

<details>
<summary>모범 답안</summary>

Next.js App Router는 **파일명 규칙(Convention)** 으로 특수한 UI를 자동 처리합니다.

| 파일 | 역할 | 내부 동작 |
|---|---|---|
| **`loading.tsx`** | 데이터 로딩 중에 보여줄 대체 UI | React `<Suspense>` 경계를 자동으로 생성하여, 서버 컴포넌트가 데이터를 가져오는 동안 이 UI를 표시합니다. |
| **`error.tsx`** | 런타임 에러 발생 시 보여줄 에러 UI | React Error Boundary를 자동으로 감싸, 하위 컴포넌트에서 에러가 발생해도 전체 앱이 깨지지 않고 이 UI를 표시합니다. `'use client'` 지시어가 필수입니다. |
| **`not-found.tsx`** | 존재하지 않는 경로 접근 시 보여줄 404 UI | `notFound()` 함수 호출이나 매칭되지 않는 URL 접근 시 이 UI를 표시합니다. |

이 파일들은 해당 폴더와 하위 경로 전체에 적용되며, 폴더 계층별로 세분화하여 다른 UI를 보여줄 수 있습니다.

</details>

### Q6. `create-next-app`으로 프로젝트를 생성할 때 Turbopack을 선택하는 이유와, 기존 Webpack과의 차이를 설명해주세요.

<details>
<summary>모범 답안</summary>

**Turbopack**은 Next.js 15부터 개발 모드(dev)에서 안정화(Stable)된 **Rust 기반 초고속 번들러**입니다.

**Webpack과의 차이:**

| 구분 | Webpack | Turbopack |
|---|---|---|
| **언어** | JavaScript | Rust |
| **개발 서버 시작** | 프로젝트가 커질수록 느려짐 | 최대 700배 빠른 시작 |
| **HMR (Hot Module Replacement)** | 상대적으로 느림 | 거의 즉시 반영 |
| **메모리 사용** | 높음 | 효율적 |

**선택 이유:**
- 개발 중 코드 변경 시 즉각적인 피드백으로 **개발 생산성**이 크게 향상됩니다.
- 대규모 프로젝트에서도 일관된 빠른 속도를 유지합니다.
- Next.js 팀이 공식적으로 밀고 있는 차세대 표준 번들러입니다.

> **참고:** Turbopack은 현재 개발 모드(`next dev`)에서 사용되며, 프로덕션 빌드(`next build`)는 여전히 Webpack을 사용합니다.

</details>
